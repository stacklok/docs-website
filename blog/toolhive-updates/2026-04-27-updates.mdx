This week, Virtual MCP Server (vMCP) is now supported locally, the ToolHive
Desktop UI installs skills directly into the AI clients you already use, and the
Registry Server surfaces version information in list views.

{/* truncate */}

## Run vMCP locally via the CLI

You no longer need a Kubernetes cluster to try vMCP. With the new local CLI
mode, you can take the MCP servers you already run on your machine and present
them to your AI client as a single endpoint, with the same aggregation, tool
routing, and optimizer that vMCP provides in production.

This is a good fit when you want to:

- Evaluate vMCP locally before rolling it out centrally via Kubernetes.
- Develop locally against several MCP servers behind one endpoint instead of
  configuring each client one server at a time.
- Cut down the tool list your AI client sees with the built-in optimizer, so
  prompts stay fast and focused.

Start with the [local vMCP CLI guide](/toolhive/guides-vmcp/local-cli). When
you're ready for a shared production deployment, the same concepts carry over to
the Kubernetes [`VirtualMCPServer`](/toolhive/guides-vmcp/quickstart) resource.

## Install skills into your AI client through ToolHive

ToolHive Desktop UI v0.32.2 makes the Skills tab the place to discover, install,
and share skills:

- **Install skills into your AI client.** Pick a skill from the bundled registry
  and install it straight into Claude Code, Cursor, or any other client ToolHive
  manages. No manual file copying or path wrangling.
- **130 skills available out of the box.** The bundled registry ships with 130
  [skills](/toolhive/concepts/skills) ready to install.
- **Connect your own skills registry.** Point ToolHive at a registry your team
  maintains to surface your organization's curated skills alongside the bundled
  ones. See [Use the registry](/toolhive/guides-ui/registry) for how to add a
  custom registry in the desktop app.
- **Package your own skills for installation.** Author a skill locally, publish
  it to a registry, and install it through the same flow as everything else, so
  internal workflows are distributed the same way as community ones. See
  [Manage skills](/toolhive/guides-registry/skills) for the publishing flow.

## Version metadata in registry list responses

The Registry Server's `ListRegistryEntries` endpoint now embeds version metadata
inline with each entry returned in the list response. Previously, clients had to
follow up with a per-entry detail request to retrieve version information,
turning catalog pages into an N+1 query pattern. With version fields included in
the list payload, UIs and registry clients can render staleness signals, "update
available" badges, and version-aware filters directly from a single paginated
response. This applies to both MCP server and skill entries served through the
registry-scoped browse endpoints.
