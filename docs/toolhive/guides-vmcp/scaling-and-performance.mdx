This guide explains how to scale Virtual MCP Server (vMCP) deployments and
documents the known scalability limits so you can plan capacity appropriately.
For MCPServer scaling, see
[Horizontal scaling](../guides-k8s/run-mcp-k8s.mdx#horizontal-scaling) in the
Kubernetes operator guide.

## Vertical scaling

Vertical scaling (increasing CPU/memory per instance) is the simplest approach
and works for all use cases, including stateful backends.

To increase resources, configure `podTemplateSpec` in your VirtualMCPServer:

```yaml
spec:
  podTemplateSpec:
    spec:
      containers:
        - name: vmcp
          resources:
            requests:
              cpu: '500m'
              memory: 512Mi
            limits:
              cpu: '1'
              memory: 1Gi
```

Vertical scaling is recommended as the starting point for most deployments.

## Horizontal scaling

Horizontal scaling (adding more replicas) can improve availability and handle
higher request volumes.

### How to scale horizontally

Set the `replicas` field in your VirtualMCPServer spec to control the number of
vMCP pods:

```yaml title="VirtualMCPServer resource"
spec:
  replicas: 3
```

If you omit `replicas`, the operator defers replica management to an HPA or
other external controller. You can also scale manually or with an HPA:

**Option 1: Manual scaling**

```bash
kubectl scale deployment vmcp-<VMCP_NAME> -n <NAMESPACE> --replicas=3
```

**Option 2: Autoscaling with HPA**

```bash
kubectl autoscale deployment vmcp-<VMCP_NAME> -n <NAMESPACE> \
  --min=2 --max=5 --cpu-percent=70
```

### Session storage for multi-replica deployments

When running multiple replicas, configure Redis session storage so that sessions
are shared across pods and survive pod restarts. Without session storage, a
request routed to a different replica than the one that established the session
will fail, and sessions are lost when pods restart.

```yaml title="VirtualMCPServer resource"
spec:
  replicas: 3
  sessionStorage:
    provider: redis
    address: redis.toolhive-system.svc.cluster.local:6379
    db: 0
    keyPrefix: vmcp-sessions
    passwordRef:
      name: redis-password
      key: password
```

The example above assumes Redis is deployed using the manifests in the
[Horizontal scaling session storage](../guides-k8s/redis-session-storage.mdx#horizontal-scaling-session-storage)
guide, which creates a Service named `redis` and a Secret named `redis-password`
in the `toolhive-system` namespace. Update `address` and `passwordRef.name` to
match your Redis Service and Secret if they differ.

:::warning

If you configure multiple replicas without session storage, the operator sets a
`SessionStorageWarning` status condition on the resource but **still applies the
replica count**. Pods will start, but requests routed to a replica that did not
establish the session will fail. Ensure Redis is available before scaling beyond
a single replica.

:::

### When horizontal scaling is challenging

Horizontal scaling works well for **stateless backends** (fetch, search,
read-only operations) where sessions can be resumed on any instance.

However, **stateful backends** make horizontal scaling difficult:

- **Stateful backends** (Playwright browser sessions, database connections, file
  system operations) require requests to be routed to the same instance that
  established the session
- Session resumption may not work reliably for stateful backends

The `VirtualMCPServer` CRD includes a `sessionAffinity` field that controls how
the Kubernetes Service routes repeated client connections. By default, it uses
`ClientIP` affinity, which routes connections from the same client IP to the
same pod:

```yaml
spec:
  sessionAffinity: ClientIP # default
```

:::warning[ClientIP affinity is unreliable behind NAT or shared egress IPs]

`ClientIP` affinity relies on the source IP reaching kube-proxy. When clients
sit behind a NAT gateway, corporate proxy, or cloud load balancer (common in
EKS, GKE, and AKS), all traffic appears to originate from the same IP - routing
every client to the same pod and eliminating the benefit of horizontal scaling.
This fails silently: the deployment appears healthy but only one pod handles all
load.

For stateless backends, set `sessionAffinity: None` so the Service load-balances
freely. For stateful backends where true per-session routing is required,
`ClientIP` affinity is a best-effort mechanism only. Prefer vertical scaling or
a dedicated vMCP instance per team instead.

:::

## Known limits and constraints

This section describes the hardcoded limits and configuration-driven constraints
in the current vMCP implementation. Use this information to plan capacity,
configure Redis, and set OS-level tuning parameters before you scale beyond a
single replica.

### Per-pod session cache

Each vMCP pod maintains a node-local LRU cache with a hard cap of **1,000
concurrent sessions**. When the cache is full, the least-recently-used session
is evicted and its backend connections are closed. Clients whose sessions are
evicted receive an error and must reconnect.

This limit is not currently configurable from the VirtualMCPServer CRD. If you
expect more than 1,000 concurrent sessions, scale horizontally by adding
replicas and configuring Redis session storage.

### Session TTL

vMCP sessions have a default TTL of **30 minutes**. A session that receives no
requests for 30 minutes is cleaned up automatically.

- **Local storage (single replica):** The session is removed from the in-memory
  store after the TTL elapses.
- **Redis storage (multi-replica):** The TTL is a sliding window. Each read
  resets the expiry via `GETEX`, so an active session is preserved indefinitely.
  A session becomes eligible for Redis eviction only after the full 30 minutes
  of inactivity.

When a session is explicitly terminated (for example, a client disconnect or an
authorization failure), vMCP issues a `DEL` to Redis immediately rather than
waiting for the TTL to expire. Abandoned sessions that are never explicitly
terminated expire automatically.

:::note

The 30-minute session TTL is a property of the vMCP server and applies
regardless of whether you use local or Redis session storage. Stateful backend
operations (tool calls, streaming responses) in progress when a session expires
are terminated without completing.

:::

### File descriptor limits

Each open backend connection consumes a file descriptor. A single vMCP pod that
aggregates many MCP backends can exhaust the pod-level file descriptor limit
before it reaches the 1,000-session cache cap.

The default Linux `nofile` limit is typically 1,024 per process. To increase it,
configure `securityContext.sysctls` or a custom `ulimit` wrapper in your pod
template:

```yaml
spec:
  podTemplateSpec:
    spec:
      containers:
        - name: vmcp
          securityContext:
            allowPrivilegeEscalation: false
      initContainers:
        - name: set-fd-limit
          image: busybox
          command: ['sh', '-c', 'ulimit -n 65536']
          securityContext:
            privileged: true
```

Alternatively, set the limit at the node level via the container runtime, or
increase it in your Kubernetes node configuration. The correct value depends on
the number of backends multiplied by the expected concurrent session count.

### Redis sizing guidance

When Redis session storage is enabled, session data is written on every new
session and refreshed on every request. Plan your Redis instance size based on
the following:

| Parameter                       | Guidance                                                                      |
| ------------------------------- | ----------------------------------------------------------------------------- |
| **Memory per session**          | Estimate 10–50 KB depending on tool count and session metadata size           |
| **Maximum concurrent sessions** | Replicas × 1,000 (capped by the per-pod LRU cache)                            |
| **Connection count**            | One connection pool per vMCP pod; size pools to match your replica count      |
| **Eviction policy**             | Use `allkeys-lru` so Redis can evict stale session keys under memory pressure |
| **Persistence**                 | Not required; session loss on Redis restart causes clients to reconnect       |

Start with a `256Mi` Redis instance for small deployments and monitor
`used_memory` as you scale. If `used_memory` exceeds 70% of `maxmemory`, add
memory or switch to a larger instance before session writes start failing.

Redis timeouts default to 5 seconds (dial), 3 seconds (read), and 3 seconds
(write). If your Redis instance is overloaded, increase these by configuring a
dedicated Redis instance rather than raising timeouts, which would mask the
underlying capacity issue.

### Stateful backends and pod restarts

vMCP is a stateless proxy: it routes MCP protocol messages to backend pods. The
backend MCP servers themselves may be stateful (for example, Playwright browser
sessions, database connections, or open file handles).

If a vMCP pod restarts or is evicted while a stateful backend session is in
progress, **the in-flight operation is lost**. The backend connection is torn
down without sending a graceful shutdown message. Clients see an error and must
reinitiate the session and retry any interrupted operations.

Redis session storage preserves the session routing table across pod restarts,
so clients can reconnect and establish a new session. However, any state held
exclusively on the backend side (open browser tabs, transaction cursors, and so
on) is not recovered. Design your workflows to tolerate reconnects, or use
vertical scaling to minimize the risk of pod restarts for workloads with
long-running stateful operations.

:::warning

Multi-replica deployments with stateful backends require that all requests from
a given session reach the same backend pod. `ClientIP` affinity provides
best-effort routing but fails silently when clients are behind NAT. For
production stateful workloads, prefer vertical scaling (one large pod) over
horizontal scaling.

:::
